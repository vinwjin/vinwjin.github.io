浏览器 MCP 方案踩坑全记录：Playwriter 主力，Playwright Bridge 备用

这篇文章记录我在 Windows + Edge 环境下，把浏览器接入 MCP 给 AI Agent 控制时踩过的坑，以及最后跑通后的可用方案。

Playwriter 主力 Playwright Bridge 备用 复用现有标签页

1. 结论先说

  • 方案一：browser-mcp（Playwriter） 当前最稳，可以复用当前已打开标签页和登录态，适合作为主力方案。
  • 方案二：官方 Playwright MCP Bridge 现在已经能用，但长会话稳定性一般，更适合作为备用方案。

2. 适用对象

本文不是只针对某一个客户端写的，而是面向所有支持 MCP 的 Agent 客户端，例如 Codex、Claude Desktop / Claude Code、WorkBuddy、Hermes，以及其他支持 STDIO MCP 或 HTTP MCP 的客户端。

  • STDIO MCP：填写启动命令、参数、环境变量
  • HTTP MCP：填写 http://127.0.0.1:PORT/mcp

3. 方案概览

方案复用当前标签页复用登录态结论
browser-mcp（Playwriter）是是主力方案
@playwright/mcp --isolated否否不适合现有登录态场景
@playwright/mcp --browser msedge --extension是是已打通，但稳定性一般

4. 方案一：Playwriter + browser-mcp

4.1 架构

AI Agent -> browser-mcp -> Playwriter Relay -> Playwriter 扩展 -> 现有 Edge 标签页

核心思路是：不新开浏览器，直接复用你已经打开的 Edge，由扩展把当前标签页暴露给 MCP。

4.2 参考链接

  • Playwriter 官网
  • Playwriter GitHub
  • Playwriter 扩展页
  • browser-mcp 仓库
  • 官方 Playwright MCP

4.3 必要环境

  1. Edge / Chrome / Chromium 之一，本文实测环境是 Windows + Edge
  2. Node.js 18+
  3. Playwriter 扩展，目标标签页要手动点成绿色激活
  4. Playwriter CLI，用于启动 relay 和排查环境
  5. playwriter-browser-mcp，它才是给 MCP 客户端接入的这一层

4.4 安装步骤

先装 Playwriter 扩展，再装 CLI，然后拉下 browser-mcp 源码构建：

npm install -g playwriter
git clone https://github.com/nicepkg/playwriter-browser-mcp.git
cd playwriter-browser-mcp
npm install
npm run build

4.5 先启动 Relay

playwriter serve --host localhost --replace

确认 19988 端口在监听后，再连 MCP。

4.6 MCP 配置

STDIO 版 JSON：

{
  "mcpServers": {
    "browser-mcp": {
      "command": "node",
      "args": ["C:\\Users\\vinwj\\playwriter-browser-mcp\\dist\\stdio.js"],
      "env": {
        "PLAYWRITER_RELAY_HOST": "127.0.0.1",
        "PLAYWRITER_RELAY_PORT": "19988",
        "PLAYWRITER_EXTENSION_ID": "profile:b87e8782196ef80c"
      }
    }
  }
}

表单版直接照抄：

  • 名称：browser-mcp
  • 类型：STDIO
  • 启动命令：node
  • 参数：C:\Users\vinwj\playwriter-browser-mcp\dist\stdio.js
  • 环境变量：
    • PLAYWRITER_RELAY_HOST=127.0.0.1
    • PLAYWRITER_RELAY_PORT=19988
    • PLAYWRITER_EXTENSION_ID=profile:b87e8782196ef80c
  • 工作目录：C:\Users\vinwj\playwriter-browser-mcp

HTTP 版：

cd C:\Users\vinwj\playwriter-browser-mcp
set MCP_PORT=3280
set PLAYWRITER_RELAY_HOST=127.0.0.1
set PLAYWRITER_RELAY_PORT=19988
set PLAYWRITER_EXTENSION_ID=profile:b87e8782196ef80c
node dist/index.js

然后在客户端中填 http://127.0.0.1:3280/mcp。

4.7 PLAYWRITER_EXTENSION_ID 最重要的坑

推荐写法：PLAYWRITER_EXTENSION_ID=profile:b87e8782196ef80c
也可以在只有一个活跃扩展时暂时留空，让 relay 自动回退。
但不要把 relay 返回的瞬时 extensionId 直接抄进配置，例如 mog5rahs_8it4bv，否则会报 Unknown extensionId。

4.8 使用前自检

  1. playwriter --version
  2. relay 是否在监听 19988
  3. 目标标签页的 Playwriter 图标是否已变绿
  4. MCP 客户端里是否写了三项环境变量

4.9 实测结果

这次我专门做了连通性验证、现有标签页复用验证、稳定性循环验证和全功能矩阵验证。

  • 能枚举现有标签页
  • 能接管当前已打开的 Edge 标签页
  • 能读取 URL / Title / 正文内容

4.10 一个容易忽略的新坑：Playwriter 的 ref 用 DOM id

在 Playwriter 这套实现里，很多交互工具的 ref，实测可以直接写 DOM id，例如 text-input、select-input、hover-target。它不是官方 Bridge 常见的 e1 / e2 / e3 那种风格。

以下能力已经实测通过：

  • browser_run_code 的代码建议写成 async (page) => ...
  • browser_console_messages 必须显式传 level，例如 {"level":"info"}

4.12 稳定性

在同一会话里连续跑 10 步循环后：

  • 固定 PLAYWRITER_EXTENSION_ID=profile:b87e8782196ef80c：10/10 成功
  • 不写 PLAYWRITER_EXTENSION_ID 自动回退：10/10 成功

这点明显好于官方 Playwright Bridge。

4.13 目前仍不建议当成绝对稳定能力的点

  • click + prompt 这一条链路还不够稳
  • network_requests 可能混入之前页面遗留请求

5. 方案二：官方 Playwright MCP Bridge

如果你坚持走官方方案，需要安装官方 Bridge 扩展，并使用：

npx @playwright/mcp@latest --browser msedge --extension

这里必须显式带 --browser msedge，只写 --extension 时，很多环境下会默认去找 Chrome。

6. 最终建议

  1. 优先用 Playwriter + browser-mcp
  2. 官方 Bridge 作为备用
  3. 不建议再把 --isolated 当主力方案

一句话总结：当前 Windows + Edge 场景下，想复用现有标签页和登录态，browser-mcp（Playwriter） 是更稳的主力解法。
